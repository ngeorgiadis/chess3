# 03 — UI/UX Specification

## UX vision

The app should feel like a calm personal chess coach, not a database spreadsheet. The UI should make the next useful action obvious.

Design qualities:

- Focused.
- Beautiful board-first layout.
- Low cognitive load.
- Fast keyboard navigation.
- Clear progress and confidence.
- Excellent dark mode.

## Information architecture

Primary navigation:

1. Today
2. Games
3. Review
4. Openings
5. Lessons
6. Exercises
7. AI Studio
8. Engines
9. Settings

## App shell

### Left sidebar

- Product logo/name.
- Primary navigation.
- Import button.
- Current engine status.
- AI provider status.

### Top command bar

- Search: games, positions, lessons, openings.
- Quick action: Import, Analyze, Create Lesson, Practice Due.
- Sync/status indicator for local jobs.

### Main content area

Adaptive based on route, with board-first layout for chess tasks.

## Screen specs

### 1. Today screen

Purpose: tell the user exactly what to do today.

Widgets:

- Today's training plan.
- Top weakness this week.
- Due opening lines.
- Due exercises.
- Recent game review queue.
- Progress streak and rating goal.

Primary CTA:

- “Start today's session.”

Interaction:

- Clicking Start opens a guided sequence: puzzles → opening recall → mistake review → endgame drill → summary.

### 2. Import modal

Input modes:

- Chess.com username or URL.
- Lichess username or URL.
- PGN file drag/drop.
- Paste PGN.
- Single game URL.

Fields:

- Platform auto-detected.
- Username.
- Date range.
- Time controls.
- Rated/casual.
- Max games.
- Include clocks/evals/opening where supported.
- Analyze after import toggle.

UX requirements:

- Show what will be fetched before fetching.
- Show expected number of archives/games when available.
- Show rate-limit friendly sequential progress.
- Allow cancellation.
- Show duplicates skipped.

### 3. Games screen

Layout:

- Left filter panel.
- Game table/list.
- Right preview board and metadata.

Columns:

- Date.
- Color.
- Opponent.
- Result.
- Time control.
- Opening.
- Accuracy/eval loss summary.
- Mistake tags.
- Analysis status.

Actions:

- Review.
- Analyze.
- Add to collection.
- Export PGN.
- Delete.

### 4. Review screen

Main layout:

- Center: chessboard.
- Right: coaching panel.
- Bottom: move timeline/eval graph.
- Left: move list and critical moments.

Coaching panel states:

1. Before mistake: “Find candidate moves.”
2. After user choice: “Compare with game move.”
3. Engine reveal: “What the engine saw.”
4. Human explanation: “What to learn.”
5. Practice conversion: “Create exercise.”

Critical moment card:

- Move number.
- Position diagram.
- User move.
- Better move.
- Eval swing.
- Theme.
- Explanation.
- Create exercise button.

### 5. Opening trainer

Views:

- Repertoire tree.
- Line practice.
- Own-game deviations.
- Model games.
- Stats.

Tree UX:

- Nodes show recall status, score, and last practiced.
- Color coding should be subtle and accessible.
- Hover previews position.
- Clicking node opens board and notes.

Practice UX:

- App plays opponent moves.
- User inputs repertoire move.
- Wrong move shows hint, not immediate punishment.
- After line ends, ask plan question.

### 6. Lessons screen

Course list:

- Active plan.
- Recommended.
- Library.
- Generated by AI.

Lesson player layout:

- Board left.
- Step content right.
- Progress stepper top.
- Notes bottom/right drawer.

Step types:

- Concept: explanation and board position.
- Demonstration: app animates moves.
- Guided question: user answers with move/text/eval.
- Puzzle: user makes move(s).
- Reflection: user writes summary.

### 7. AI Studio

Purpose: create validated lessons from input text, notes, PGN, or game mistakes.

Tabs:

- Source input.
- Extraction outline.
- Generated JSON.
- Validation results.
- Preview lesson.

UX requirements:

- Display copyright/use reminder: “Use source material you are authorized to transform.”
- The agent must first produce outline; user can edit before generation.
- Validation errors must be actionable.
- Preview must match final lesson player.

### 8. Engines screen

Cards for installed engines:

- Name/version.
- Path.
- Protocol.
- Status.
- Options.
- Profiles using this engine.

Actions:

- Add engine.
- Verify engine.
- Configure options.
- Benchmark.
- Remove.

### 9. Settings

Sections:

- Profile and rating goals.
- Platforms.
- AI providers/local models.
- Storage/export.
- Privacy.
- Appearance.
- Keyboard shortcuts.

## Chessboard interaction requirements

- Drag/drop pieces.
- Click source then target.
- Keyboard move entry.
- Arrow keys for move navigation.
- Flip board.
- Coordinates toggle.
- Highlight last move.
- Legal move hints toggle.
- Blunder heatmap optional.
- Support arrows and square highlights in lessons.

## Visual design tokens

No Chess.com-proprietary board/piece assets.

Suggested tone:

- Neutral dark background.
- Elegant board with accessible contrast.
- Calm typography.
- Minimal badges.
- Evaluation shown as a bar, not a scary number by default.

## Accessibility

- Keyboard-only navigation.
- Screen-reader labels for board coordinates and pieces.
- High-contrast mode.
- Reduced motion mode.
- Color should never be the only way to communicate correctness.

## Empty states

### No games imported

Message: “Import your games to build a personalized training plan.”
CTA: “Import games.”
Secondary CTA: “Try sample course.”

### No engine installed

Message: “Add a UCI engine to analyze games locally.”
CTA: “Add engine.”
Secondary CTA: “Use lesson mode without engine.”

### No AI configured

Message: “AI lesson generation is optional. You can still study and analyze games.”
CTA: “Configure AI provider.”

## Microcopy principles

Bad: “Centipawn loss exceeded threshold.”
Good: “Your move allowed a tactic because the knight on f3 was pinned.”

Bad: “You blundered.”
Good: “This was the critical mistake. Train this pattern.”

## Delight moments

- Weekly “mistakes fixed” report.
- Before/after opening stats.
- “You solved a position you missed in a real game.”
- Beautiful generated lesson previews.
